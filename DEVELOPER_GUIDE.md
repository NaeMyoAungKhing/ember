# Ember — Developer Guide

*Ember — Systematic Review Triage. Search, rank, screen, hand off to ASReview.*

A side-project roadmap for a click-based tool that does a full systematic-review search workflow in the browser.

> **Files that must stay together:** `SR_Triage_Tool.html` + `Flame.mp4` (the animated flame marker, referenced relatively). Keep them in the same folder / same GitHub repo. Everything else is generated by the tool.

---

## 1. What it is

`SR_Triage_Tool.html` is a web page (the flame video is its only logo/marker) — no install, no server, no account. You double-click it and it opens in your browser. It reproduces what was done by hand during the build:

1. Build a search from concept keyword groups
2. Query **OpenAlex** (a free, open scholarly database — ~250M works, CC0 data)
3. De-duplicate and reconstruct abstracts
4. Relevance-rank so on-topic papers float up and noise sinks
5. Screen each record (include / maybe / exclude) with one click
6. Export a worksheet CSV, an included-papers CSV, and an open-access download hub

Everything runs in the browser using plain JavaScript and the public OpenAlex API. That's the whole trick — **no backend needed**, because OpenAlex allows cross-origin browser requests.

## 2. How to run & edit it

- **Run:** double-click `SR_Triage_Tool.html` (or drag it into Chrome).
- **Edit:** open it in any text editor (VS Code is ideal, free). The HTML, CSS, and JavaScript are all in the one file, in that order. Change something, save, refresh the browser.
- **Publish (optional):** you already have a GitHub Pages site (naemyoaungkhing.github.io). Drop this file into a repo and it's instantly a live web app at a URL you can share.

## 3. The one thing to understand: the OpenAlex API

Everything depends on one URL pattern. Paste this into a browser to see raw data:

```
https://api.openalex.org/works?filter=title_and_abstract.search:(neuroleadership)&per-page=5
```

Key pieces the tool uses:

- **`filter=title_and_abstract.search:(...)`** — your query. Combine with commas for AND-ed filters: `,type:article`, `,language:en`, `,publication_year:2000-2026`.
- **Boolean inside the search:** `(a OR b) AND (c OR d)`. Phrases in quotes: `"SCARF model"`.
- **Pagination (cursor):** ask for `&cursor=*`, then each response gives `meta.next_cursor` — keep passing it until it's null. `&per-page=200` is the max.
- **`select=`** — only pull the fields you need (faster): `id,doi,title,publication_year,authorships,abstract_inverted_index,best_oa_location,open_access`.
- **Abstracts are "inverted"** — a `{word: [positions]}` map (for copyright reasons). The `deinvert()` function rebuilds readable text from it.
- **Open-access PDF:** `best_oa_location.pdf_url`, and `open_access.oa_status` (gold/green/diamond/hybrid/bronze/closed).
- **Polite pool:** add `&mailto=you@email.com` for faster, more reliable responses.

Full docs: https://docs.openalex.org

## 4. Code map (where to change things)

The JavaScript is organised into small functions — find them by name:

| Function | Does |
|---|---|
| `buildFilter()` | turns your concept boxes + filters into the OpenAlex `filter` string |
| `preview()` | shows the match count before fetching |
| `run()` | fetches all pages (cursor loop), de-duplicates, calls scoring |
| `deinvert()` | rebuilds an abstract from the inverted index |
| `rescore()` | the relevance scoring rule (core +5, context +2, noise −3) — **edit this to change ranking logic** |
| `render()` | draws the results table; handles filter/sort/hide |
| `setDec()` / `autoTriage()` | screening decisions |
| `exportCSV()` / `downloadHub()` / `saveSession()` / `loadSession()` | the export + save buttons |

Inputs persist between refreshes via `localStorage` (bottom of the script).

## 5. The ASReview connection (the key step)

The tool's job is **search + rank + first-pass triage**; ASReview's job is **AI-assisted screening** (active learning). The clean, reliable bridge between them is a **file hand-off** — no fragile live integration needed, and it works for anyone.

**How it works now (built in):**
- The **"Download RIS for ASReview"** button exports your records as a standard `.ris` file. ASReview imports RIS natively (Create → File → drop the file).
- RIS fields written: `TY` (type), `TI` (title), `AU` (each author), `PY` (year), `JO` (journal), `AB` (abstract), `DO` (doi), `UR` (url). Your screening labels are carried in an `N1` note (`screening_label: relevant/irrelevant/maybe`) so they aren't lost.
- There's also an **"ASReview-ready CSV"** export with the exact columns ASReview recognises (`title, abstract, authors, doi, publication_year, url`) — use this if you prefer CSV.

**Two ways to use the bridge:**
1. *Whole search → ASReview:* export everything with "Download RIS for ASReview", screen the full set in ASReview's active-learning interface. (Recommended — let the AI do the heavy lifting.)
2. *Pre-triaged → ASReview:* triage in the tool first, then export "included + maybe only" for a focused second pass.

**Advanced (optional) — live connection to a running ASReview:**
Today's manual workflow talked directly to ASReview's local API while `asreview lab` was running. You *can* replicate this in the tool, but it's fragile (needs ASReview running, CORS between `file://` and `localhost`, and a created project), so it's not the default. The endpoints, if you want to experiment:
- `GET  http://localhost:5000/api/projects` — list projects
- `GET  http://localhost:5000/api/projects/<id>/get_record` — next record to screen
- `POST http://localhost:5000/api/projects/<id>/record/<record_id>` with body `record_id=<id>&label=1|0` — label a record

Stick with the RIS hand-off for anything you share with others.

## 6. Roadmap — features to add, easiest first

**✅ Already built:** OpenAlex search + cursor pagination · de-duplication · relevance ranking · click + keyboard screening (`i` include, `m` maybe, `x` exclude, `↑/↓` move) · keyword highlighting · **multi-database import & merge** (drop in Scopus/WoS/PubMed `.ris` or `.csv`, auto-dedup against OpenAlex) · live PRISMA counter · RIS + ASReview-CSV export · open-access download hub · save/load session.



**Quick wins**
- [ ] **PRISMA counter** — a box showing: identified → after de-dup → screened → included. You already compute these numbers; just display them and let the user export a PRISMA-style summary.
- [ ] **More than two concept boxes** — generalise `buildFilter()` to loop over an array of concept groups so you can do 3–4 concepts.
- [ ] **Per-record "reason" field** — a text box on each row, saved in the CSV, for documenting why you excluded something (examiners love this).
- [ ] **Highlight matched terms** in the title/abstract so you can see *why* a paper ranked high.

**Medium**
- [ ] **RIS / BibTeX export** — so the included set imports straight into Zotero, EndNote, or ASReview. (RIS is a simple text format: `TY  - JOUR`, `TI  - title`, `AU  - author`, `DO  - doi`, `ER  -`.)
- [ ] **CSV/RIS import** — let users bring in Scopus or Web of Science exports and merge + de-duplicate them with the OpenAlex set (multi-database coverage = stronger review).
- [ ] **De-duplication by fuzzy title** — current de-dup is exact DOI/title; add normalised-title matching to catch near-duplicates across databases.
- [ ] **Inclusion-criteria checklist** — a small form (population, study type, date) that auto-flags records.

**Ambitious**
- [ ] **Active-learning ranking** — a lightweight in-browser model (e.g. TF-IDF + logistic regression in JS) that re-ranks unscreened records after each decision, like ASReview. This is the big one and a genuine contribution.
- [ ] **Citation chasing** — use OpenAlex `referenced_works` / `cited_by_api_url` to pull a paper's references and citations (snowballing).
- [ ] **Backend + accounts** — only if you want saved projects across devices; not needed for a personal tool.

## 7. Testing tips

- Start every change with a **small `Max records` (e.g. 40)** so fetches are instant.
- Use **Preview count** before a big fetch.
- Open the browser **DevTools console (F12)** to see errors.
- Verify a query by pasting the API URL directly in a browser tab — if the raw JSON looks right, the tool will too.

## 8. Gotchas worth knowing

- **PDFs don't all auto-download** — many publishers render PDFs inline or block bots; the tool links to them instead of forcing downloads.
- **Acronym collisions** — "SCARF" also means a gene (SCARF1) and clothing. Anchor ambiguous acronyms to context terms in your scoring/noise lists.
- **Browser downloads go to your Downloads folder** unless you change Chrome's download location.
